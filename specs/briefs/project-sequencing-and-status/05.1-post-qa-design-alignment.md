# Delegation brief — Slice 05.1: Post-QA alignment, agent surfacing, numbered folders

## Intent

Three improvements to the pipeline defaults, all discovered during slice 05 signoff review:

1. **Numbered project subfolders** — `01-design/`, `02-slices/`, `03-briefs/`, `04-tasks/`, `05-qa/`. Subfolders currently sort alphabetically (briefs before design, qa before slices) which makes the project folder hard to read. Numbering in pipeline order fixes that. All path references in command files need updating.

2. **Agent surfacing at tasks_ready** — When `/design` Phase 7 completes and outputs the tasks_ready gate, it should show the proposed agent team (derived from task `agent_type` fields), state whether it's a single agent or team, and ask for confirmation. The human may want to update task files before running `/implement`.

3. **Post-QA design alignment** — After QA passes, before advancing to `signoff_review`, the agent reviews the design doc and all future (unimplemented) slice files for anything that needs updating based on what was learned. Summarizes changes, applies them, leaves them uncommitted alongside the implementation diff.

## Observable outcomes

**Numbered subfolders:**
```
Given /design creates a new project
When project artifacts are written
Then design docs go to 01-design/, slices to 02-slices/,
     briefs to 03-briefs/, task files to 04-tasks/slice-{NN}/,
     QA reports to 05-qa/
And ls .orchestration/projects/{id}/ shows folders in pipeline order
```

**Agent surfacing at tasks_ready:**
```
Given /design Phase 7 has just created task files
When the tasks_ready gate is shown
Then output includes the unique agent_type values from all task files
And explicitly states "single agent: {type}" or "team: {types}"
And asks: "Is this the right team, or do you want to adjust task agent_types first?"
And waits — /implement should not be run until the human has seen this
```

**Post-QA design alignment — no updates needed:**
```
Given QA has passed for a slice
When the agent reviews design doc and future slices
Then output includes "no design or slice updates needed"
And advances to signoff_review
```

**Post-QA design alignment — updates found:**
```
Given QA has passed and a future slice has a stale assumption
When the agent reviews future slices
Then the agent summarizes what needs updating and why
And updates the affected slice files and/or design doc
And those changes are left uncommitted alongside the implementation diff
And the signoff_review output lists what was changed
```

## Constraints

**Must:**
- Update all path references in `defaults/commands/design.md`, `slice.md`, `spec.md`, `implement.md`, `qa.md` to use numbered subfolder paths
- New paths: `01-design/design-{NN}.md`, `02-slices/{NN}-{slug}.md`, `03-briefs/{NN}-{slug}.md`, `04-tasks/slice-{NN}/{NN}-{slug}.md`, `05-qa/slice-{NN}-qa-report.md`
- Agent surfacing added to design.md Phase 7 tasks_ready gate output
- Post-QA review added to qa.md as a phase between QA verification and signoff_review advance
- Post-QA changes left uncommitted — no git add/commit

**Must not:**
- Change any logic other than paths in the folder rename — slicing, spec, breakdown logic unchanged
- Block the tasks_ready gate indefinitely — agent surfaces team and asks, human answers, pipeline continues

**Preferences:**
- Agent surfacing format: one line per type with task count, then "single agent" or "team of N"
- Post-QA summary: brief — one bullet per changed file, one sentence why

**Escalation triggers:**
- A future slice's scope overlaps significantly with what was just implemented — stop and ask before modifying it

## Done definition

1. All five command files use numbered subfolder paths consistently
2. tasks_ready gate in design.md shows agent team, labels single/team, asks for confirmation
3. qa.md has a post-QA review phase that summarizes and applies alignment changes before signoff_review
4. Post-QA changes are not committed (behavior rule enforced)
5. `ls .orchestration/projects/{id}/` shows numbered folders in pipeline order
